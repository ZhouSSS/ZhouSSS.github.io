WARNING

以上ip和port配置是与运行容器时的配置相关 1.若为更改http和ssh端口，则无需更改。其中修改gitlab_rails与external_url，在复制git仓库的clone地址时，直接相关，当复制clone地址时，则会变为 ssh://git@外部ip:外部ssh端口/用户/项目.git 和 http://外部ip:外部http端口/用户/项目.git

2.如果更改了外部http端口，external_url ‘http://192.168.137.11:8080’ 配置项改成带端口的，改成非80端口，则必须添加 nginx[‘listen_port’] = 80 ，因为external_url带端口则会让容器内的nginx监听改的端口，但运行容器时是 -p 8080:80，则外部一定访问不到Gitlab服务

3.nginx[‘listen_port’] 此项在更改外部http端口时，必须添加，否则外部访问不到Gitlab服务

三、Jenkins部署与配置

1.运行Jenkins容器

2.配置Jenkins

2.1获取管理员密码，网页访问并输入密码解锁Jenkins

2.2 安装插件

1）直接选安装推荐插件，等待安装完毕 2）创建管理员用户 admin 3）安装其他插件 系统管理 → 插件管理 → 可选插件搜索框，搜索并安装以下插件：

  • Gitlab Gitlab API
  • Docker
  • Publish Over SSH

2.3创建Jenkins任务

2.3.1 创建freestyle项目

输入项目名称xxx，选择freestyle项目。名字最好能与Gitlab的项目对应

2.3.2 配置freestyle项目

2.3.2.1 配置源码管理

源码管理选择Git Repository URL：此处可以填入Gitlab项目的ssh或者http两种地址区别为 1）填入http地址，则Jenkins凭证选择Gitlab创建项目的用户或有权限提交的用户和其密码连接Github 2）填入ssh地址，则在Jenkins凭证中用户填入git，私钥选择与Gitlab项目配对的私钥，且需要在容器内使用私钥连接一次后，添加目标服务器的指纹到known_hosts文件中，网页配置生效。

私钥挂载或复制进容器，进入容器执行后，可添加目标服务器的指纹到known_hosts文件：

2.3.2.2 配置触发器
  1. 勾选Build when a change is pushed to GitLab … 同时记得 GitLab webhook URL
  2. 下翻点击Generate，生成token
  3. 切到Gitlab，在项目中Setting → Webhooks，填入上述URL和token，勾选适当的触发条件(Push events等)，下拉最后点击确定，生成webhooks，点击edit后，能看到历史触发的详细信息。点击test，可以发送对应的触发事件，用来手动测试联通Jenkins的流水线或其他任务
2.3.2.3 配置构建

这里主要是配置C++/CMake构建项目。所以 build step 勾选 执行shell 输入如下：

本地自行准备好编译环境的镜像，另外此处是Jenkins容器挂载目录/opt/jenkins_home:/var/jenkins_home，相应的Jenkins拉取的Gitlab项目会在容器内的/var/jenkins_home/workspace/项目名 ，也就是宿主机的/opt/jenkins_home/workspace/项目名，选择运行编译容器时，将Gitlab项目目录由宿主机挂载进编译容器，且工作目录进入项目目录下，此处需开发人员将构建命令自行准备好，且随项目一同提交进Gitlab。执行过程中，编译完成的可执行文件和库都会在宿主机的/opt/jenkins_home/workspace/项目名 目录下，取决于 build.sh的构建细节。后续可将构建的产物继续打包或上传

Jenkins的构建步骤里，build step可以是一个或多个，因此可以拆分多步，也可以一步构建完，然后使用docker build打包

build step比较自由，主要服务于最终的目的，能达成目的即可
